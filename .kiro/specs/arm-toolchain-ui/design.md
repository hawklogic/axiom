# Design Document: ARM Toolchain UI

## Overview

This design document describes the architecture and implementation approach for the ARM Toolchain UI in Axiom IDE. The UI consists of five main components integrated into the existing IDE layout, providing a comprehensive interface for embedded ARM development.

## Component Specifications

### 1. ArmToolchainPanel.svelte

Main container component that provides tab navigation between the five sub-panels.

```typescript
interface ArmToolchainPanelProps {
  // No props - uses stores for state
}

interface TabDefinition {
  id: 'settings' | 'build' | 'visualizer' | 'compliance' | 'memory';
  label: string;
  icon: string;
  shortcut?: string;
}
```

**State Management:**
- Uses `armToolchainStore` for toolchain state
- Uses `complianceStore` for compliance state
- Local state for active tab

**Behavior:**
- Renders tab bar with icons and labels
- Conditionally renders active tab content
- Shows loading overlay during operations
- Handles keyboard shortcuts for tab switching

Configuration panel for ARM toolchain selection and MCU settings.

  settingsScope: 'project' | 'global';

## Correctness Properties

### P1: Toolchain Selection Persistence
For any toolchain selection made by the user, the selection SHALL be persisted and restored on IDE restart.

**Test Strategy**: Property-based test that selects random toolchains, simulates restart, and verifies selection is restored.

### P2: Build Output Completeness
For any build operation, all stdout and stderr output from the toolchain SHALL be captured and displayed in the build output panel.

**Test Strategy**: Property-based test that runs builds with various inputs and verifies output completeness.

### P3: Error Navigation Accuracy
For any clickable error in the build output, clicking SHALL navigate to the exact file and line indicated in the error message.

**Test Strategy**: Property-based test that generates errors at random locations and verifies navigation accuracy.

### P4: Memory Usage Calculation
For any successful build, the memory usage percentages displayed SHALL accurately reflect the ratio of used to total memory for each region.

**Test Strategy**: Property-based test that verifies `displayed_percentage == (used / total) * 100` for all regions.

### P5: Compliance Mode State Consistency
For any sequence of compliance mode toggles, the UI state SHALL match the backend state.

**Test Strategy**: Property-based test that performs random toggle sequences and verifies UI/backend consistency.

### P6: Traceability Matrix Completeness
For any project with requirement annotations, all annotated requirements SHALL appear in the traceability matrix.

**Test Strategy**: Property-based test that generates projects with random annotations and verifies matrix completeness.

### P7: Settings Scope Isolation
Project-specific settings SHALL NOT affect global settings, and global settings SHALL NOT override project-specific settings when both exist.

**Test Strategy**: Property-based test that modifies settings in both scopes and verifies isolation.
